# 医美智能面诊系统

AI 辅助的医美面诊、报告生成与机构管理系统。当前项目已经按 1Panel 容器编排部署场景整理，默认外部访问端口为 `15000`。

## 当前进度

- 已完成机构、员工、管理员登录与基础管理，支持机构内容值提示、员工归属机构、项目类别与提示词预设管理。
- 已完成图片上传持久化，报告列表与详情页可稳定预览上传图片，避免上传后图片丢失。
- 已完成分析报告时间精确到秒，报告状态支持 `analyzing`、`image_processing`、`completed`、`failed`。
- 已完成异步分析流程：`/api/analysis` 先返回文本分析结果，AI 完整报告图片区域显示占位；后台生成提拉线标注图和完整 PNG 后，详情页轮询刷新展示。
- 已完成接口配置拆分：文本分析接口和提拉线图片接口分开管理，可分别选择 OpenAI 或 Grok/xAI；配置项会根据供应商切换。
- 已完成图片生成方式切换：支持基于原图编辑 `edit` 和参考原图生成新图 `generate` 两种模式。
- 已完成 AI 图片传输调整：图片输入和图片生成结果按 base64/data URL 处理，不再把远程图片 URL 写入报告。
- 已完成 AI 完整报告 PNG 合成：系统负责合成雷达图、综合评分、整体评价、问题诊断、问题详解、改善方案、效果预期对比和提拉线标注图，避免 AI 直接生成报告文字导致乱码或分数失真。
- 已完成中文字体兼容：Docker 镜像安装 Noto CJK 字体，完整报告图片中的中文可正常显示。
- 已完成提拉线图片提示词更新：以原图为主，只允许调整为正面面诊视角；点位为淡红色，提拉线为白色大间隔细虚线，不改变背景，不遮挡五官。
- 已完成 1Panel 部署包：包含 `Dockerfile`、`docker-compose.yml`、`.env.example` 和启动脚本，容器内包含 OpenSSL 与 Prisma 运行所需依赖。

## 脱敏说明

- 本仓库不提交真实 `.env`、`1panel-medspa-deploy/.env`、数据库、上传图片、构建产物、依赖目录和部署压缩包。
- `.env.example` 与 `1panel-medspa-deploy/.env.example` 只保留占位配置，请部署时自行填写真实 Key、JWT 密钥和供应商地址。
- 已对常见 OpenAI、xAI Key 形态和自定义接口域名进行扫描，当前代码中仅发现占位值、环境变量名和业务字段名。

## 1Panel 部署

1. 使用项目根目录的 `1panel-medspa-deploy` 文件夹，或现有的 `1panel-medspa-deploy.zip` 上传到 1Panel。
2. 在 1Panel 中创建容器编排，选择该目录内的 `docker-compose.yml`。
3. 复制 `.env.example` 为 `.env`，至少修改：
   - `JWT_SECRET`
   - `OPENAI_API_KEY` 或 `XAI_API_KEY`
   - `OPENAI_BASE_URL` 或 `XAI_BASE_URL`
   - 文本和图片模型配置
4. 启动后访问 `http://服务器IP:15000`。

容器数据默认挂载：

- `/app/data`：SQLite 数据库
- `/app/public/uploads`：上传图片与生成报告图片

## 关键配置

- `APP_PORT`：容器外部端口，当前为 `15000`。
- `AI_ANALYSIS_PROVIDER`：文本分析供应商，支持 `openai` 或 `grok`。
- `AI_IMAGE_PROVIDER`：提拉线图片供应商，支持 `openai` 或 `grok`。
- `AI_IMAGE_MODE`：图片生成方式，`edit` 表示基于原图编辑，`generate` 表示参考原图生成新图。
- `DATABASE_URL`：SQLite 数据库路径，容器内建议使用 `file:/app/data/medspa.db`。

## 已知事项

- Next.js 构建时，部分依赖 cookie 的 API 路由会出现 `DYNAMIC_SERVER_USAGE` 提示，这是动态接口的构建期提示；当前构建流程可正常结束。
- 当前后台图片生成是进程内异步任务，适合单机部署；如果容器重启，正在生成的图片任务可能需要重新发起。
- AI 提拉线效果仍强依赖图片模型能力和提示词服从度，已优先约束“原图背景不变、白色虚线、淡红点位、医学美学提拉方向”。

## 生产化改造路线图

### P0：上线前必须补齐

- **密钥与配置安全**：生产环境必须使用独立 `.env`，替换默认 `JWT_SECRET`、API Key 和模型地址；禁止在后台页面明文回显 Key，建议保存前加密，展示时只显示后四位。
- **默认账号治理**：上线前必须修改或禁用默认 `admin/admin123`、`demo/123456`，新增首次启动强制改密或初始化管理员流程。
- **数据备份**：当前 SQLite 数据挂载在 `/app/data`，生产部署需配置 1Panel 定时备份，至少覆盖数据库和 `/app/public/uploads`。
- **反向代理超时**：分析接口和图片生成链路可能超过默认网关超时，Nginx/1Panel 代理建议设置 `proxy_read_timeout`、`proxy_send_timeout`、`proxy_connect_timeout` 不低于 180 秒。
- **健康检查**：为容器增加 `/api/health` 或等价探活接口，检查应用进程、数据库连接、上传目录读写权限。
- **日志保留**：容器日志需要配置滚动策略，避免 AI 请求失败、图片生成失败等日志撑满磁盘。

### 架构与任务队列

- **图片生成任务外置**：当前图片生成使用进程内异步任务，后续应迁移到 Redis + BullMQ、RabbitMQ 或数据库任务表，支持任务持久化、重试、失败恢复和多实例消费。
- **分析流程拆分**：建议拆成文本分析任务、提拉线图片任务、完整 PNG 合成任务三个阶段，每个阶段记录开始时间、结束时间、耗时、错误码和重试次数。
- **任务状态可观测**：前端报告详情页除占位图外，应展示“排队中 / 生成中 / 失败可重试 / 已完成”的明确状态。
- **幂等处理**：重复点击分析、浏览器刷新、后台重试时，应通过 `reportId` 或请求指纹避免重复扣积分、重复创建报告。
- **横向扩展准备**：如果未来多容器部署，上传文件和报告图片不能依赖单容器本地磁盘，需要统一存储和可共享任务队列。

### 数据库与存储

- **数据库升级路径**：SQLite 适合单机轻量部署；生产规模扩大后建议迁移到 PostgreSQL 或 MySQL，保留 Prisma migration 作为结构变更来源。
- **迁移管理**：从 `prisma db push` 过渡到 `prisma migrate deploy`，避免生产库结构被非审计方式直接改写。
- **对象存储**：上传原图、提拉线图、完整报告 PNG 建议迁移到 S3、MinIO、阿里云 OSS 或腾讯云 COS，数据库只保存对象 Key、大小、MIME、hash 和访问策略。
- **文件生命周期**：为临时图、失败报告图、长期归档报告设置清理策略，避免存储无限增长。
- **数据归档**：报告、积分流水、AI 调用记录应分层归档，避免主表过大影响后台查询。

### 安全与合规

- **访问控制**：现有管理员、员工权限需要继续细化为 RBAC，例如接口配置、机构管理、员工管理、报告查看、积分管理分别授权。
- **机构隔离**：所有报告、图片、员工、积分流水查询都必须强制带机构边界，避免跨机构数据泄露。
- **接口频控**：登录、上传、分析、图片生成、后台配置保存接口需要增加 IP、账号、机构维度的限流。
- **上传安全**：限制图片 MIME、大小、分辨率和文件扩展名；落盘前重新编码图片，降低伪装文件和异常图片解析风险。
- **审计日志**：记录登录、配置修改、Key 更新、积分变动、报告删除、重新生成图片等高风险操作。
- **隐私保护**：面部照片属于敏感个人信息，建议增加数据保留期限、删除能力、下载审计和最小化访问策略。
- **传输安全**：生产环境必须启用 HTTPS，Cookie 设置 `Secure`、`HttpOnly`、`SameSite`，并按域名收紧跨域策略。

### AI 能力与成本控制

- **供应商降级策略**：文本分析和图片生成应支持主备供应商，主供应商失败或超时时自动切换备用供应商，并记录最终使用链路。
- **成本计量**：记录每次 AI 调用的模型、输入大小、输出大小、耗时、状态、预估成本和对应机构，支撑后续计费或风控。
- **提示词版本化**：提示词、模型、供应商、图片模式和报告合成版本需要写入报告元数据，保证历史报告可追溯。
- **提拉线可控性增强**：优先引入人脸关键点检测或视觉定位模型，先确定苹果肌、法令纹、下颌线、太阳穴、耳前等锚点，再让图片模型做轻量医疗可视化标注。
- **失败兜底**：图片模型失败时保留系统合成报告，提拉线区域展示失败原因和重试入口，不影响文本报告交付。
- **内容安全**：对 AI 输出做 JSON schema 校验、字段补全、敏感词和医疗承诺过滤，避免出现 `undefined`、乱码或夸大疗效描述。

### 可观测性与运维

- **结构化日志**：日志应包含 `requestId`、`reportId`、`organizationId`、`employeeId`、供应商、模型、耗时和错误码。
- **指标监控**：至少监控接口 P95/P99 延迟、分析成功率、图片生成成功率、队列堆积、AI 供应商错误率、磁盘使用率、数据库大小。
- **告警策略**：图片生成连续失败、AI 供应商超时升高、磁盘空间不足、数据库备份失败、登录失败激增时需要告警。
- **错误追踪**：接入 Sentry 或同类工具，区分前端渲染异常、API 异常、后台任务异常和第三方接口异常。
- **发布回滚**：保留上一个可用 Docker 镜像标签和数据库备份点，支持 1Panel 中快速回滚。

### 性能与用户体验

- **上传优化**：上传前可做客户端压缩和方向矫正，服务端保留原图和分析用压缩图，减少 AI 请求体积。
- **报告渲染优化**：完整 PNG 合成可以缓存，报告详情页只请求已有图片；大图加载增加缩略图或渐进加载。
- **长任务体验**：分析提交后立即展示文本报告，图片区域显示进度、预计等待时间和失败重试按钮。
- **移动端适配**：报告详情、拍照上传、后台配置页面需要补充移动端布局验证，避免文字挤压和按钮换行失控。
- **批量操作**：机构报告多时，需要增加搜索、筛选、分页、批量导出和批量删除。

### 测试与发布流程

- **单元测试**：继续覆盖 AI 响应解析、base64 图片提取、报告状态流转、报告图片合成、配置选择逻辑。
- **接口测试**：补充登录、上传、分析、报告轮询、后台配置、机构隔离等 API 集成测试。
- **端到端测试**：使用 Playwright 覆盖员工完整面诊流程、管理员配置流程、图片生成失败重试流程。
- **Docker 冒烟测试**：每次发版前构建镜像，启动容器，执行数据库初始化、登录、上传、创建分析报告的最小链路验证。
- **CI/CD**：建立 GitHub Actions、Gitea Actions 或本地流水线，执行依赖安装、类型检查、测试、构建、镜像构建和部署包生成。
- **版本管理**：为每次部署打 Git tag，记录数据库迁移版本、镜像标签、README 变更和回滚说明。

### 建议实施顺序

1. 先补齐 P0 上线项：密钥治理、默认账号、备份、超时、健康检查、日志滚动。
2. 再做任务队列和对象存储，解决异步图片生成和多实例部署的基础问题。
3. 然后做安全审计、RBAC、机构隔离测试和上传安全。
4. 接着完善 AI 成本计量、提示词版本化和提拉线关键点控制。
5. 最后补齐 CI/CD、E2E、监控告警和正式发布回滚流程。
